Unified Dropdown Component — Approach

How should the shared <Dropdown> component be built?

A

Wrap shadcn DropdownMenu

Build a thin wrapper around the shadcn/ui DropdownMenu primitive (Radix UI under the hood). Gets focus-trap, keyboard navigation, portal rendering (no z-index wars), and ARIA for free. Style it to match the company selector look.

Pros

  • Keyboard nav & focus trap built in
  • Portal = no overflow/z-index issues
  • Accessible by default
  • Already in the project (shadcn installed)

Cons

  • Portal makes it harder to style relative to trigger
  • More abstraction — harder to customise fully
B

Custom React component, same pattern as company selector

Build a <Dropdown> component that mirrors the AppShell company selector — trigger button, absolute-positioned panel, click-outside + Escape to close, chevron rotation. Simple, predictable, on-brand.

Pros

  • Matches existing sidebar dropdown exactly
  • Full styling control
  • Simple to understand and extend

Cons

  • Manual z-index management
  • Must hand-wire keyboard nav & ARIA
  • Click-outside logic duplicated from AppShell
C

Headless Popover (Radix) with full custom UI

Use Radix Popover primitive directly (no shadcn layer) and build the entire visual layer from scratch. Maximum control, portal included, but more boilerplate than option A.

Pros

  • Portal rendering
  • Full visual control
  • Good accessibility primitives

Cons

  • More setup than A
  • Adds Radix dependency directly (already indirect via shadcn)
