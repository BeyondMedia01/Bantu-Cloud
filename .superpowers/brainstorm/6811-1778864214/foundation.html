Bantu — Foundation Layer Design

This is what Prompt #1 to Lovable must establish. Everything else builds on this.

1. Database Schema (Supabase Postgres)

Multi-Tenancy Tables
  • clients — top-level tenant (bureau/org)
  • companies — belongs to client
  • sub_companies — optional under company
  • branches — under company
  • departments — under company
Auth & Access Tables
  • users — extends Supabase auth.users
  • roles — custom roles per company
  • role_permissions — module × action matrix
  • user_roles — user → role assignment
  • licenses — subscription tier + enabled modules
People Tables
  • employees — master record
  • grades — salary bands
  • employee_documents
  • employment_history
Payroll Tables
  • payroll_runs
  • payslips
  • transaction_codes
  • employee_transactions
  • loans + loan_repayments

2. Row Level Security (RLS) Strategy

Every table with company data has RLS policies that enforce:
auth.uid() → user_companies → company_id = row.company_id

Platform admins bypass via a service role key in Edge Functions only — never exposed to the client. Client admins see all companies under their client_id. Company users see only their assigned company. Employees see only their own rows.

4. The 15 Modules (Phase 1 = Tier 1 only)

PEOPLE TIME_LEAVE PAYROLL COMPLIANCE REPORTS SETTINGS RECRUITMENT (Phase 2) PERFORMANCE (Phase 2) EXPENSES (Phase 2) ONBOARDING (Phase 3) TRAINING (Phase 3) ASSETS (Phase 3) SUCCESSION (Phase 4) SURVEYS (Phase 4) ANALYTICS (Phase 4)

Each module has 7 actions: VIEW · EDIT · DELETE · APPROVE · EXPORT · RUN · CONFIGURE

5. Cloudflare Security Layer

Cloudflare sits in front of Supabase:
WAF rules → Rate limiting (auth: 5 req/15min, API: 200 req/15min) → DDoS protection → Custom domain routing → Supabase project URL hidden from public
