# Bantu Platform — Lovable Full-Stack Rebuild Design

**Date:** 2026-05-15
**Status:** Approved
**Scope:** Full rebuild of Bantu Payroll & HR SaaS using Lovable (React frontend) + Supabase (Postgres + Auth + Edge Functions) + Cloudflare (security layer)

---

## 1. Overview

Bantu is a Zimbabwe-focused Payroll & HR SaaS with native multi-currency support (USD/ZiG), deep statutory integrations (ZIMRA, NSSA, NEC), and a multi-tenant architecture serving payroll bureaus, SMEs, and enterprises.

The rebuild replaces the existing React/Node.js/Prisma stack with:
- **Frontend:** React (generated by Lovable)
- **Backend:** Supabase (Postgres + Row Level Security + Edge Functions for payroll calculations)
- **Security:** Cloudflare (WAF, rate limiting, DDoS protection, hiding Supabase URL)
- **Email:** Supabase Auth built-in for invitations; Resend for transactional notifications (leave approvals, payroll run status)
- **Biometric integration:** Deferred to Phase 2

---

## 2. Architecture

```
Browser (React)
    │
    ▼
Cloudflare (WAF + Rate Limiting + DDoS)
    │
    ▼
Supabase
    ├── Auth (JWT, invitations, sessions)
    ├── Postgres (multi-tenant data + RLS)
    └── Edge Functions (payroll calculations, PDF generation, exports, notifications)
```

### Rate Limits (enforced at Cloudflare)
- Auth endpoints: 5 requests / 15 min
- API endpoints: 200 requests / 15 min

## 3. Multi-Tenancy Model

Three-level hierarchy:

```
Client (payroll bureau / org)
  └── Company
        ├── SubCompany (optional)
        ├── Branch
        └── Department
              └── Employee
```

All data is scoped to `company_id`. RLS policies enforce this at the database level — no application-level filtering required.

## 5. Row Level Security Strategy

Every company-scoped table has RLS policies. Three policy patterns cover all four roles:

```sql
-- Pattern 1: COMPANY_USER / EMPLOYEE — single company
CREATE POLICY "company_scoped"
ON employees FOR ALL
USING (
  company_id = (SELECT company_id FROM users WHERE id = auth.uid())
);

-- Pattern 2: CLIENT_ADMIN — all companies under their client
CREATE POLICY "client_admin_scoped"
ON employees FOR ALL
USING (
  company_id IN (
    SELECT id FROM companies
    WHERE client_id = (SELECT client_id FROM users WHERE id = auth.uid())
  )
);

-- Pattern 3: EMPLOYEE self-service — own rows only (via user_id FK)
CREATE POLICY "employee_self_service"
ON payslips FOR SELECT
USING (
  employee_id = (SELECT id FROM employees WHERE user_id = auth.uid())
);
```

Access rules by role:
- **PLATFORM_ADMIN:** Uses service role key in Edge Functions only — never client-side
- **CLIENT_ADMIN:** Pattern 2 — sees all companies under their `client_id` (company_id is NULL in users table)
- **COMPANY_USER:** Pattern 1 — sees only their assigned `company_id`. Note: Pattern 1 grants full table access within the company; RBAC module/action enforcement for COMPANY_USER is done at the frontend and Edge Function layer, not Postgres RLS.
- **EMPLOYEE:** Pattern 3 (applied to `payslips`, `leave_requests`, `leave_balances`, `attendance_records`, `employee_documents`) — sees only rows linked to their `employees.user_id`. All Pattern 3 tables resolve via the same join: `employee_id IN (SELECT id FROM employees WHERE user_id = auth.uid())`. Never compare `auth.uid()` directly against `employee_id` — `employees.id` is not a user UUID.

## 7. Payroll Engine (Edge Functions)

All statutory calculations run server-side in Supabase Edge Functions. Core logic ported from existing `backend-v2` TypeScript.

Each payroll run snapshots the `exchange_rate_id` and `tax_table_id` at processing time — these are frozen into the run record for audit immutability.

### Statutory Calculations
- **PAYE:** ZIMRA tax bands from `tax_tables`, 3 methods: `FDS_AVERAGE`, `FDS_FORECASTING`, `NON_FDS`
- **AIDS Levy:** 3% of PAYE
- **NSSA:** Employee + employer contributions with thresholds
- **NEC:** Industry council levies
- **SDF / WCIF / ZIMDEF:** Additional statutory contributions
- **Dual-currency:** Per-employee USD/ZiG split percentages applied to gross and net
- **Medical Aid Credit:** Informational display only, excluded from totals

### Edge Functions
| Function | Prompt | Purpose |
|----------|--------|---------|
| `POST /functions/v1/calculate-payroll` | 4 | Full run calculation, writes payslips + line items. Idempotent: deletes existing payslips/line_items for the run before recalculating. Only callable when run status is PREVIEW or SUBMITTED. |
| `POST /functions/v1/generate-payslip-pdf` | 5 | HTML-rendered dual-currency payslip PDF. Payslip reads (list/view) use the Supabase client directly against the `payslips` table — no Edge Function needed for reads. |
| `POST /functions/v1/export-bank-eft` | 6 | CBZ / Stanbic / Fidelity EFT formats |
| `POST /functions/v1/export-zimra-p2` | 6 | ZIMRA monthly return (includes ZIMDEF) |
| `POST /functions/v1/export-nssa-p4a` | 6 | NSSA monthly return |
| `POST /functions/v1/export-zimra-it7` | 6 | Annual IT7 (P16) certificates |
| `POST /functions/v1/send-notification` | 1 | Email via Resend — leave approvals, run status, invitations |

## 8. Lovable Prompt Sequence

### Prompt 1 — Foundation
**Delivers:** Working login + empty dashboard for all 4 roles + email notifications wired

Covers: Supabase project setup · Full DB schema (all tables + RLS, all 3 policy patterns) · 4-role auth flow (login, invite, company switcher) · Navigation shell with role-based sidebar · Default "Full Access" role seeded for CLIENT_ADMIN · `send-notification` Edge Function (Resend) built once here, called by Prompts 3+ for leave/payroll notifications · Cloudflare proxy config

### Prompt 2 — People Module
**Delivers:** Full employee database

Covers: Employee CRUD (full lifecycle + termination, all statutory ID fields) · Bulk CSV import · Org hierarchy · Salary grades + NEC min/max · Document management

### Prompt 3 — Time & Leave Module
**Delivers:** Full leave + attendance (no biometrics)

Covers: Leave policy config · Leave request workflow (submit → approve → reject, email notification on status change via `send-notification` from Prompt 1) · Leave balance tracking · Monthly accrual via `pg_cron` inside Postgres (not an Edge Function) · Shift configuration · Manual attendance entry · Public holiday management

### Prompt 4 — Payroll Engine
**Delivers:** End-to-end payroll run with statutory deductions + line item breakdown

Covers: Transaction codes (with structured conditions schema) · Payroll calendar (cut-off + pay date + locking) · Run lifecycle (Preview → Submit → Approve → Process) · Exchange rate + tax table snapshot on run · Dual-currency USD/ZiG · PAYE + AIDS Levy + NSSA + NEC + SDF + WCIF + ZIMDEF via `calculate-payroll` Edge Function · `payroll_line_items` written per run · Loan repayment auto-deduction

### Prompt 5 — Payslips & Loans
**Delivers:** Payslips employees can view and download

Covers: `generate-payslip-pdf` Edge Function · Dual-currency payslip PDF with line item breakdown · Print / download · Employee self-service payslip view · Loan management (create, repayment schedule) · Back pay processing (`back_pay_entries` table)

### Prompt 6 — Compliance & Statutory Exports
**Delivers:** Full statutory compliance exports

Covers: ZIMRA IT7 / P2 (inc. ZIMDEF) · NSSA P4A · Tax table management with effective dates · Bank EFT files (CBZ, Stanbic, Fidelity) · Motor vehicle benefit

### Prompt 7 — Reports Module
**Delivers:** Full reporting suite (20+ report types)

Covers: CSV / PDF / XLSX export · Payroll variance (uses `payroll_line_items`) · Leave reports · Loan reports · Headcount · Journal entries · Pension exports · Payroll trends

### Prompt 8 — RBAC, Platform Admin & Settings
**Delivers:** Full admin + platform control plane

Covers: Custom role builder (15 modules × 7 actions) · User invitation + management · License key management + seat tracking · Audit logging · System settings (currency rates, holidays, NSSA thresholds) · Platform admin: client lifecycle + global config

## 10. Success Criteria

- All 4 user roles can log in and see their correct navigation
- A CLIENT_ADMIN managing 10 companies sees all of them via the company switcher; a COMPANY_USER sees only their own
- A COMPANY_USER without PAYROLL permission cannot access the payroll run page (frontend guard + Edge Function rejection — note: Postgres RLS for COMPANY_USER is company-scoped only; module-level enforcement is application-layer)
- A full payroll run produces correct PAYE, NSSA, AIDS Levy, ZIMDEF for a dual-currency employee and writes one `payroll_line_item` row per transaction code
- Payslips are downloadable as PDF with USD and ZiG figures
- ZIMRA P2 and NSSA P4A exports match the required statutory format
- RLS cross-tenant test: a seed script queries company B's employees using company A's JWT and receives 0 rows
- Payroll run for 500 employees completes within 30 seconds via Edge Function
